# 拾流订阅中心产品规划

## Goal

规划“拾流 / Shiliu”：一个面向开发者与信息重度用户的订阅中心，统一管理文章、博客、新闻、播客与音频节目，并支持阅读、收听、收藏、稍后处理、搜索、AI 摘要和 Obsidian 导出。

## Confirmed Facts

- 产品中文名：拾流。
- 产品英文名：Shiliu。
- 初始定位：RSS 阅读器与播客集合体。
- 目标使用者：开发者 / 信息重度用户。
- 内容类型：文章、博客、新闻、播客、音频节目。
- MVP 的内容条目类型只落成两类：`text` 和 `audio`；文章、博客、新闻归入 `text`，播客单集和音频节目单集归入 `audio`；更细的文章 / 博客 / 新闻分类后置。
- 核心能力候选：订阅管理、阅读、收听、收藏、稍后处理、搜索、AI 摘要、Obsidian 导出。
- 规划方法：先列出功能全集，再划分哪些进入 MVP、哪些进入后续迭代。
- MVP 必须包含基础搜索、单条 AI 摘要、单条 Obsidian Markdown 导出，以保留拾流相对普通订阅阅读器的差异化。
- MVP 输入来源限定为 RSS feed URL、Podcast RSS feed URL 和 OPML 一次性批量导入；网页 URL、Newsletter、YouTube、社交媒体和视频平台来源进入后续迭代。
- MVP 订阅源类型只落成 `rss` 和 `podcast` 两类；包含播客 RSS 语义或音频 enclosure 的订阅源归 `podcast`，其余结构化 feed 归 `rss`；更细的来源分类后置。
- 添加订阅源和 OPML 导入时，MVP 只按规范化后的 feed URL 判断订阅源重复，并对 `feeds.feed_url` 建唯一约束；标题、站点链接和描述不参与去重。
- 手动添加订阅源时，必须先成功抓取并解析 feed 才创建订阅源；URL 无法抓取或解析失败时不保存订阅源，只返回错误。
- OPML 导入中的单个订阅源也必须先成功抓取并解析 feed 才创建；失败项计入失败数量，不创建空订阅源。
- MVP OPML 导入只读取 feed URL；忽略 OPML 原有文件夹 / 分组层级，不自动创建拾流文件夹，也不自动把订阅源分配到文件夹。
- feed URL 规范化只做安全且可预测的处理：trim、去掉 fragment、scheme / host 小写、默认端口归一；跨 URL canonical 探测和相似源合并后置。
- 拾流的核心对象统一称为“内容条目”；文章、博客、新闻、播客单集和音频节目单集都是内容条目的不同类型。
- 内容条目采用“未处理 / 已完成”作为主状态；“稍后处理”和“收藏”作为可叠加标记。
- 内容条目的“已完成”只由使用者手动标记；阅读 / 收听进度不自动改变处理状态。
- MVP 只持久化音频播放进度，不持久化文本阅读位置。
- MVP 音频播放器支持播放 / 暂停、进度条拖动、继续播放、15 秒快进 / 后退和倍速播放；播放队列、章节和离线下载作为后续迭代。
- MVP 的 AI 摘要只基于已有文本：文本型内容条目用正文 / 摘要 / 描述 / 标题；音频型内容条目用标题、节目描述和 show notes；音频转写后置。
- MVP 内容条目必须保留原始 feed 文本字段，并在抓取入库边界对正文字段做 HTML 净化后双存（原始 + 净化后）；阅读渲染使用净化后 HTML，`available_text` 从净化后内容去标签生成纯文本，作为搜索、AI 摘要和 Obsidian 导出的统一输入；`available_text` 生成优先级为 `content` → `show_notes` → `description/summary` → `title`。
- MVP HTML 净化使用单一全局 `bluemonday` 白名单策略，发生在抓取解析后、入库前；禁止按来源 / 按站点适配净化规则；按来源适配净化后置。
- MVP 必须支持手动摘要；自动摘要也进入 MVP，但必须由使用者配置开启，不能默认强制对所有抓取内容生成摘要。
- MVP 的自动摘要配置只支持全局开关和内容类型范围；按订阅源、文件夹、标签、关键词或复杂规则配置后置。
- MVP 自动摘要只对配置开启后新抓取入库的内容条目生效；不回填已有历史内容条目，开启配置时不批量补摘要。
- MVP 的 AI 摘要输出为固定结构化 Markdown：TL;DR、要点、对开发者 / 信息重度用户的价值 / 行动、原文信息；摘要正文固定使用简体中文，保留必要英文术语、代码名、产品名和链接原文。
- MVP 的 AI 摘要服务由使用者自带 API Base URL、API Key 和模型名配置；拾流不内置官方服务端密钥。
- MVP AI 服务协议只支持 OpenAI-compatible、非流式 Chat Completions 接口；Claude、Gemini、Ollama、本地模型或其他服务必须通过 OpenAI-compatible endpoint / proxy 接入；原生多供应商适配后置。
- MVP 保存 AI 服务配置时只做格式校验，不强制发起模型测试请求；提供可选“测试配置”动作；自动摘要开启前只要求配置字段存在，不要求最近测试成功。
- MVP 将 AI API Key 保存在单实例数据库中并仅由服务端使用；前端保存后只显示“已配置”状态，不回显完整密钥；数据库字段加密、外部 secret manager 和环境变量式密钥管理后置；导出内容、搜索结果、错误信息和日志不得包含 API Key。
- 如果音频型内容条目的可用文本不足，MVP 必须诚实提示无法生成可靠摘要，而不是生成幻觉式摘要。
- MVP 的 AI 摘要失败后不自动重试；失败状态记录原因并允许使用者手动重试。可用文本不足属于终态失败，不应重试。
- MVP 每个内容条目只保留一个当前 AI 摘要及其状态；状态为 `none` / `pending` / `success` / `failed` / `insufficient_text`；手动重试成功后覆盖旧失败或旧摘要，不保留摘要版本历史。
- 自动摘要只处理 `none` 状态的内容条目；不会自动覆盖 `success`，也不会自动重试 `failed` 或 `insufficient_text`。
- 手动摘要允许在 `none` / `failed` / `success` 状态触发，成功后覆盖当前摘要或错误；`pending` 时返回正在生成；`insufficient_text` 在 MVP 中不可重试。
- MVP 的 Obsidian 导出只做单条内容条目的 Markdown 复制或下载；导出包含标签和订阅源文件夹信息，但只作为普通 Markdown 元信息，不做 Obsidian frontmatter；导出不强制生成 AI 摘要，始终包含 `AI 摘要` 区块，并按当前摘要状态写入摘要内容或“未生成 / 正在生成 / 生成失败 / 可用文本不足”；导出只包含 `available_text` 前 2000 个字符作为可用文本摘录，截断时提示打开原文链接查看完整内容；自动同步、Obsidian 插件、双链、模板和 vault 深度集成后置。
- MVP 不做产品内整库备份 / 恢复、全部数据导出或一键恢复；单实例数据库 / volume 备份由部署文档说明并交给部署层处理。
- MVP 是个人单用户产品；首版支持一个用户账户用于登录、会话和访问保护，不做注册开放、多用户数据隔离、团队成员或云端多用户同步。
- MVP 首次访问进入一次性初始化流程创建唯一用户账户；创建成功后初始化入口永久关闭，之后只能登录已有账户。
- MVP 单人用户账户使用 `username + password` 登录；邮箱、昵称和 profile 不进入 MVP 用户账户模型，避免暗示邮箱验证、邮件通知、找回密码或 SaaS 个人资料能力。
- MVP 支持已登录状态下修改密码；登录页不提供 Web 端“忘记密码”流程；首版不规划 CLI 重置密码能力。
- MVP 不承诺产品内密码恢复；忘记唯一用户账户密码时，只能通过备份恢复或重置实例数据后重新首次初始化，恢复密钥、环境变量临时重置和其他恢复机制后置。
- MVP 登录会话采用 JWT Bearer Token；登录成功返回 access token，前端通过 `Authorization: Bearer <token>` 调用 API；token 默认 30 天有效；JWT 签名密钥由使用者自行修改 yml 配置；手动退出由前端删除 token；不做 refresh token、服务端 session 表、“记住我”选项、多设备会话管理、强制全部设备下线、可配置会话策略、JWT 密钥轮换或生产环境密钥强制校验。
- MVP 做登录失败限速 / 锁定所需的最小记录，用于阻止无限暴力尝试；连续 5 次失败后锁定登录 15 分钟，成功登录后清空失败计数；只按唯一用户账户记录失败次数和锁定到期时间，不做 IP / 设备维度限速、登录历史、安全审计页面、安全通知或设备列表。
- MVP 不做 2FA、TOTP、Passkey 或硬件密钥；账号安全只覆盖强密码输入（最少 12 个字符，不强制大小写 / 数字 / 符号组合）、bcrypt 密码哈希（cost 12）、30 天会话、手动退出和失败限速 / 锁定；不做可配置 cost、Argon2 或 scrypt。
- MVP 不内置 HTTPS / TLS 或证书管理；公网 VPS 部署必须由使用者自管反向代理或平台提供 TLS。
- MVP 是本地 / VPS / 自托管单实例 Web App；有后端、SQLite 文件数据库、RSS 抓取、AI 摘要和鉴权能力；不做公开无保护服务。
- MVP 使用 SQLite 文件数据库并持久化在部署 volume 中；PostgreSQL、数据库多后端支持、横向扩展和高并发数据库架构后置。
- MVP 数据库迁移必须通过显式迁移流程执行；使用 `golang-migrate/migrate` 和成对 SQL migration 文件，由 Nunu `cmd/migration` 入口执行；不得依赖运行时自动建表或裸 Gorm AutoMigrate 隐式推断结构变化。
- MVP 主部署方式为 Docker Compose：同一镜像启动 `server` 与 `task` 两个服务，共享 SQLite 数据 volume；源码本地运行只作为开发方式，不作为主要使用者部署路径。
- MVP 技术方向采用 Go 后端 + React 前端；当前规划讨论只展开后端边界，React 前端细节后续单独讨论；不采用 Next.js 全栈单体。
- MVP Go 后端采用 Nunu 作为脚手架，使用 Gin、Gorm、Wire、Viper、Zap 这套结构；拾流必须改为 SQLite-only、无 Redis、同一镜像双服务部署，不能沿用 Nunu 示例中的 MySQL / Redis 运行假设。
- MVP 后端 API 采用 REST JSON，统一前缀为 `/api/v1`；沿用 Nunu 模板统一响应结构 `{ "code": <int>, "message": "...", "data": ... }`，成功响应 HTTP 200 且 `code` 为 0，失败响应由业务 `code` / `message` 表达并用 HTTP status 表达错误类别；列表分页采用 `page + pageSize` 页码分页（`page` 从 1 开始），默认 `pageSize=20`，最大 `pageSize=100`，`data` 内含 `items` 和 `page:{page,pageSize,total}`；不做 GraphQL、tRPC、多版本 API 或把 OpenAPI/Swagger 作为产品功能。
- 当前 Nunu 模板将定时任务定义为独立 `cmd/task` 应用入口，由 `internal/server/task.go` 中的 `TaskServer` 使用 `gocron` 调度；长驻 job 与定时任务分属 `internal/job` 和 `internal/task`，订阅源后台抓取和自动摘要应按 Nunu 定时任务结构设计。
- MVP 后端分层中 `internal/task` 只做定时调度编排；订阅源抓取、OPML 导入后抓取、手动刷新、自动摘要和手动摘要的业务逻辑必须放在 service 层，并被 HTTP handler 与 `cmd/task` 共享。
- MVP 的订阅源更新机制支持手动刷新全部订阅源、手动刷新单个订阅源和全局后台定时抓取；添加新订阅源后立即抓取一次；全局后台抓取默认 60 分钟，并可在设置中选择关闭、30 分钟、60 分钟、6 小时或 24 小时；不做每源自定义抓取规则。
- MVP 必须防止同一个订阅源并发抓取；如果同源抓取已在进行中，新的同源抓取请求必须跳过并返回“已在抓取中 / 已跳过”语义；不同订阅源的并发策略由实现选择，复杂全局队列和并发池后置。
- MVP 订阅源抓取状态只在订阅源记录上保留当前 / 最后状态字段：`fetch_status`（`idle` / `fetching` / `success` / `failed`）、`fetch_started_at`、`last_fetched_at`、`last_fetch_error`；不建抓取历史表，不提供抓取历史列表或趋势分析。
- MVP 不抓取原网页全文；文本型内容条目只使用 RSS feed 提供的标题、链接、摘要、描述、正文片段或全文。
- MVP 文本阅读只做基础渲染和打开原文；高亮、批注、阅读主题、字体设置和阅读模式优化后置。
- MVP 的基础搜索只做关键词搜索和基础过滤；向量语义搜索、相似内容查找、自动主题聚类和问答式检索后置。
- MVP 基础搜索采用 SQLite FTS5 做关键词搜索；索引字段包括内容条目标题、订阅源名称、`available_text` 和当前 AI 摘要 Markdown；有关键词时搜索结果默认按相关性排序，相关性相同时按内容条目发布时间倒序；无关键词、只有过滤条件时按内容条目发布时间倒序；结构化过滤仍使用普通字段 / 关联表过滤；过滤条件只支持单值，多个过滤条件之间使用 AND；MVP 不做发布时间范围过滤、多选过滤、OR 条件组、保存搜索条件、中文分词插件、拼音搜索、模糊纠错或语义搜索。
- 内容收件箱和搜索结果共享同一套过滤语义：内容类型、处理状态、内容标记、标签和文件夹；内容收件箱不带关键词并按发布时间倒序，搜索页可带关键词，有关键词时按相关性排序。
- MVP 中标签用于组织内容条目，文件夹用于组织订阅源。
- MVP 的标签 / 文件夹只做创建、重命名、删除、单条分配和过滤；批量编辑、自动打标、嵌套文件夹和 OPML 文件夹自动映射后置。
- MVP 删除标签只删除标签本身及其与内容条目的关联，内容条目保留；删除文件夹只删除文件夹本身，原属于该文件夹的订阅源置为无文件夹（`folder_id` 置空），订阅源及其内容条目保留；与删除订阅源级联删除内容条目形成对比，标签 / 文件夹只是组织维度。
- MVP 首版导航视图为：内容收件箱、稍后处理、收藏、已完成、订阅源、标签、搜索、设置。
- MVP 内容收件箱、稍后处理、收藏和已完成都是同一内容条目列表的预设过滤视图，复用同一套查询、分页、排序、单值 AND 过滤和条目操作；内容收件箱默认 `processing_status=unprocessed`，稍后处理预设 `mark=later`，收藏预设 `mark=favorite`，已完成预设 `processing_status=completed`；使用者可在视图内追加内容类型、标签、文件夹等过滤，但不提供专属工作流或仪表盘。
- MVP 功能优先级依次为：订阅源管理 / 抓取 / 内容收件箱；文本阅读 / 音频收听 / 处理状态 / 稍后处理 / 收藏；基础搜索 / 标签 / 文件夹过滤；手动 AI 摘要 / AI 服务配置；Obsidian 单条 Markdown 导出；自动摘要；OPML 导入。
- 内容收件箱默认按内容条目发布时间倒序；缺失或异常发布时间时回退到抓取时间。
- 首次抓取或 OPML 导入时，每个订阅源最多保存最近 50 条历史内容；后续抓取只新增未见过的新内容条目。
- MVP 的后续抓取采用 insert-only：对同一订阅源内已经存在的内容条目，不更新标题、描述、正文、show notes 或 `available_text`；更新已有条目和摘要过期检测后置。
- 删除订阅源时，MVP 必须同时删除该订阅源产生的历史内容条目及其派生数据；取消订阅但保留历史内容后置。
- MVP 只在同一订阅源内去重；同源内内容条目使用 `dedupe_key` 去重，生成优先级为 feed item `guid` → `link` → `title + published_at` 稳定哈希；数据库对 `(feed_id, dedupe_key)` 建唯一约束；跨订阅源重复或相似内容合并后置。
- 仓库当前没有产品 README、ADR 或实现代码可回答产品边界问题；根 `CONTEXT.md` 已记录拾流领域词汇。

## Requirements

- 拾流必须帮助使用者统一管理多个内容来源，而不是只做单一 RSS 阅读或单一播客播放器。
- 拾流必须同时覆盖文本内容与音频内容的消费场景。
- 拾流必须以“内容条目”统一承载文本与音频内容，并允许不同类型保留自己的消费字段。
- MVP 的内容条目类型必须只分为 `text` 和 `audio` 两类；不得把文章、博客、新闻作为独立内容类型落入首版模型。
- 拾流必须支持使用者对内容进行后续处理：收藏、稍后处理、搜索、AI 摘要、导出到 Obsidian。
- 内容条目的处理进度必须与价值判断分离：未处理 / 已完成表达处理进度；稍后处理 / 收藏表达用户意图或价值判断。
- 处理状态必须由使用者手动改变；阅读 / 收听进度只能作为消费进度，不能自动把内容条目标记为已完成。
- MVP 必须保存音频播放进度以支持继续收听；MVP 不要求保存文本阅读位置。
- AI 摘要必须基于内容条目已有文本生成；当文本不足时必须明确失败原因，不能假装理解音频或缺失正文。
- 内容条目必须保留原始 feed 文本字段：`title`、`description`、`content`、`show_notes`，按 feed 原样存储；同时必须在入库边界对正文字段做 HTML 净化并存储净化后字段供渲染；必须生成去除 HTML 标签的纯文本 `available_text`，供基础搜索、AI 摘要和 Obsidian 导出复用；MVP 不得只保存单一 `content` 字段而丢失来源语义。
- HTML 净化必须在抓取解析后、入库前完成，使用单一全局 `bluemonday` 白名单策略；MVP 不得按来源 / 按站点编写净化适配规则；原始 HTML 必须保留以支持未来对存量重新净化。
- AI 摘要失败后必须记录失败原因并允许使用者手动重试；MVP 不得自动重试失败摘要，且不得重试可用文本不足这类终态失败。
- 每个内容条目必须只保留一个当前 AI 摘要及其状态；搜索和 Obsidian 导出只使用当前摘要；MVP 不得保留摘要版本历史、多模型摘要对比或旧摘要回看。
- 自动摘要必须只处理 `none` 状态的内容条目；不得覆盖已有成功摘要，不得自动重试失败摘要或可用文本不足摘要。
- 手动摘要必须允许使用者在 `none`、`failed` 和 `success` 状态下主动触发；成功后覆盖当前摘要或错误；`pending` 时必须返回正在生成；`insufficient_text` 必须作为 MVP 终态不可重试。
- 手动摘要必须始终可用；自动摘要必须是可配置能力，由使用者选择是否开启。
- 自动摘要配置必须限制为全局开关和内容类型范围，避免在 MVP 中引入规则引擎。
- 自动摘要必须只处理配置开启后新抓取入库的内容条目；MVP 不得在开启自动摘要时批量回填已有历史内容。
- AI 摘要必须输出固定结构化 Markdown，以便直接展示、搜索和进入 Obsidian 导出内容；摘要正文必须固定使用简体中文，并保留必要英文术语、代码名、产品名和链接原文。
- AI 摘要服务必须由使用者配置外部 API Base URL、API Key 和模型名；MVP 不得内置官方服务端密钥、公共配额或平台级计费。
- MVP AI 服务协议必须限制为 OpenAI-compatible、非流式 Chat Completions；不得在首版引入 provider 类型、原生 Claude / Gemini / Ollama / 本地模型协议适配或流式响应处理。
- 保存 AI 服务配置时必须只做格式校验；MVP 不得强制测试请求成功后才允许保存配置；必须提供一个可选“测试配置”动作。
- 自动摘要开启前只要求 AI 服务配置字段存在；MVP 不得要求最近测试成功、定期健康检查通过或强制模型可用性门禁。
- AI API Key 必须只在服务端使用并保存于单实例数据库；前端不得回显完整密钥，只能展示已配置状态；导出内容、搜索结果、错误信息和日志不得泄露 API Key；MVP 不要求数据库字段加密、外部 secret manager 或环境变量式密钥管理。
- Obsidian 导出必须以 Markdown 为稳定边界；MVP 不要求直接写入使用者本地 vault。
- Obsidian 导出不得强制生成 AI 摘要，也不得因为 AI 服务不可用、摘要失败或文本不足而阻止导出。
- Obsidian 导出必须保留内容条目的标签和订阅源文件夹语境，但不得在 MVP 中引入 frontmatter 或模板系统。
- Obsidian 导出必须始终包含 `AI 摘要` 区块：`success` 写入当前摘要 Markdown，`none` / `pending` / `failed` / `insufficient_text` 分别写明未生成、正在生成、生成失败或可用文本不足。
- Obsidian 导出的可用文本摘录必须来自 `available_text` 前 2000 个字符；如果被截断，必须追加“已截断，请打开原文链接查看完整内容”语义提示；MVP 不提供全文导出开关、可配置截断长度或按内容类型不同上限。
- MVP 不得提供产品内整库备份 / 恢复、全部数据导出或一键恢复；部署文档必须说明由部署层备份单实例数据库 / volume。
- MVP 必须引入单人用户账户用于登录、会话和访问保护；“使用者”表示产品操作者，“用户账户”表示单人部署中的身份对象。
- 首次访问必须进入一次性初始化流程创建唯一用户账户；创建成功后必须关闭初始化入口，之后不得再通过 Web 界面创建第二个用户账户。
- 单人用户账户必须使用 `username + password` 作为登录凭据；MVP 用户账户模型不得要求邮箱、昵称或 profile 字段，也不得提供邮箱验证、邮件通知、邮箱找回密码或个人资料编辑能力。
- MVP 用户账户模型必须只保留鉴权必需字段：`id`、`username`、`password_hash`、登录失败计数、锁定到期时间、创建时间和更新时间。
- MVP 必须支持已登录状态下修改密码；登录页不得提供 Web 端忘记密码流程；MVP 不规划 CLI 重置密码能力。
- MVP 不得承诺产品内密码恢复；忘记唯一用户账户密码时，恢复边界必须限制为备份恢复或重置实例数据后重新首次初始化。
- 登录会话必须使用 JWT Bearer Token；登录成功后返回 30 天有效的 access token，API 请求必须使用 `Authorization: Bearer <token>`；JWT 签名密钥由使用者自行修改 yml 配置；手动退出由前端删除 token；MVP 不得引入 refresh token、服务端 session 表、“记住我”、多设备会话管理、强制全部设备下线、可配置会话策略、JWT 密钥轮换或生产环境密钥强制校验。
- 登录保护必须记录失败限速 / 锁定所需的最小信息，以阻止无限暴力尝试；连续 5 次登录失败后必须锁定登录 15 分钟，成功登录后必须清空失败计数；MVP 只按唯一用户账户记录失败次数和锁定到期时间，不得提供 IP / 设备维度限速、登录历史、安全审计页面、安全通知或设备列表。
- MVP 不得引入 2FA、TOTP、Passkey 或硬件密钥；账号安全边界必须限制为强密码输入（最少 12 个字符，不强制大小写 / 数字 / 符号组合）、bcrypt 密码哈希（cost 12）、30 天会话、手动退出和失败限速 / 锁定；MVP 不得引入可配置 hash cost、Argon2 或 scrypt。
- MVP 不得内置 HTTPS / TLS 终止、证书签发或证书续期管理；公网 VPS 部署必须要求使用者通过自管反向代理或平台能力提供 TLS。
- MVP 必须以单实例数据边界运行，避免在 VPS 或自托管场景中暴露公开无保护共享实例；单人用户账户不得扩展为多用户、团队或多租户体系。
- MVP 必须使用 SQLite 文件数据库并持久化在部署 volume 中；不得在 MVP 中引入 PostgreSQL、数据库多后端支持、横向扩展或高并发数据库架构。
- 数据库结构变化必须通过 `golang-migrate/migrate` 执行成对 SQL migration 文件，且迁移必须先于 `server` / `task` 正常运行；MVP 不得依赖裸 Gorm AutoMigrate、server/task 启动时的隐式迁移或自写无版本迁移 runner。
- MVP 必须以 Docker Compose 作为主要使用者部署方式，采用同一镜像启动 `server` 与 `task` 两个服务并共享 SQLite 数据 volume；源码本地运行只作为开发方式，不作为主要部署路径。
- 订阅源类型必须只分为 `rss` 和 `podcast` 两类；订阅源类型用于订阅源展示、过滤和默认消费入口，不替代内容条目自身的 `text` / `audio` 类型判断。
- 订阅源重复判断必须只基于规范化后的 feed URL；数据库必须对 `feeds.feed_url` 建唯一约束；MVP 不得用订阅源标题、站点链接或描述判断重复。
- 订阅源创建必须以成功抓取并解析 feed 为前置条件；手动添加失败不得落库，OPML 导入中的失败项也不得创建空订阅源。
- feed URL 规范化必须限制为 trim、去掉 fragment、scheme / host 小写、默认端口归一；MVP 不做跨 URL canonical 探测、feed autodiscovery 或相似源合并。
- MVP 后端必须使用 Go；前端方向为 React，但当前后端规划不得展开 React 组件、状态管理或 UI 技术细节；不得采用 Next.js 全栈单体作为后端形态。
- Go 后端必须以 Nunu 为脚手架，保留其 handler / service / repository / middleware / cmd 分层思路，并使用 Gin、Gorm、Wire、Viper、Zap；必须将数据层改为 SQLite-only，且不得引入 Redis、MySQL、PostgreSQL 或数据库多后端适配。
- 后端 API 必须采用 REST JSON，并统一挂载在 `/api/v1` 前缀下；MVP 不得引入 GraphQL、tRPC、多版本 API，OpenAPI/Swagger 只能作为开发辅助而非产品功能。
- REST JSON 响应必须沿用 Nunu 模板统一结构 `{ "code": <int>, "message": "...", "data": ... }`：成功响应 HTTP 200 且 `code` 为 0，失败响应由业务 `code` 和 `message` 表达并用 HTTP status 表达错误类别；分页列表必须在 `data` 内包含 `items` 和 `page`，并使用 `page + pageSize` 页码分页（`page` 从 1 开始），默认 `pageSize=20`，最大 `pageSize=100`，`page` 返回 `page`、`pageSize`、`total`。
- 订阅源后台抓取和自动摘要必须按 Nunu 定时任务结构实现：通过 `cmd/task` 独立入口启动 `TaskServer`，由 `gocron` 调度 `internal/task` 中的任务逻辑；不得自定义一套与 Nunu 模板冲突的 scheduler 结构。
- `internal/task` 必须只负责调度编排，不得承载核心业务逻辑；订阅源抓取、OPML 导入后抓取、手动刷新、自动摘要和手动摘要必须复用 service 层能力，供 HTTP handler 与 `cmd/task` 共同调用。
- 订阅源必须能被手动刷新；MVP 必须同时支持手动刷新全部订阅源和手动刷新单个订阅源，也必须支持一个全局后台定时抓取机制，以形成真实订阅体验；全局抓取间隔必须可由使用者在固定选项中配置。
- 同一订阅源的抓取必须互斥，避免添加、OPML 导入、手动刷新和后台定时抓取之间重复请求或状态覆盖；MVP 不要求实现复杂全局队列、重试系统或跨实例锁。
- 订阅源抓取状态必须限制为当前 / 最后一次完成结果所需字段；MVP 不得保存每次抓取历史、错误趋势或审计日志。
- 删除订阅源必须删除该订阅源产生的内容条目及派生数据，包括摘要、播放进度、内容标记和标签关联；MVP 不得保留无订阅源归属的孤儿内容条目。
- MVP 不得为了补足文章正文而抓取原网页全文；正文获取边界必须与结构化 feed 输入保持一致。
- 文本阅读能力必须限制为基础渲染 feed 可用文本和打开原文，不得在 MVP 中引入高亮、批注、阅读主题或字体设置。
- 基础搜索必须能按关键词查找内容条目的标题、订阅源名称、可用文本和已生成 AI 摘要，并支持按内容类型、处理状态、内容标记、标签和文件夹做单值过滤；多个过滤条件之间必须使用 AND 语义。
- 内容收件箱和搜索结果必须共享同一套结构化过滤语义；内容收件箱不接受关键词，默认按内容条目发布时间倒序；搜索页可接受关键词，有关键词时按相关性排序。
- 基础搜索关键词匹配必须使用 SQLite FTS5；结构化过滤仍基于普通字段和关联表完成；MVP 不得引入发布时间范围过滤、多选过滤、OR 条件组、保存搜索条件、中文分词插件、拼音搜索、模糊纠错或语义搜索。
- 标签必须用于内容条目；一个内容条目可拥有多个标签。
- 文件夹必须用于订阅源；一个订阅源最多属于一个文件夹。
- 标签和文件夹的 MVP 能力必须限制为创建、重命名、删除、单条分配和过滤。
- 删除标签必须只删除标签本身及其与内容条目的关联，不得删除内容条目；删除文件夹必须只删除文件夹本身并把原属于该文件夹的订阅源置为无文件夹，不得删除订阅源或其内容条目；标签 / 文件夹删除是组织维度删除，与删除订阅源的内容级联删除明确区分。
- 创建标签 / 文件夹必须是独立动作；给内容条目分配标签只接受已存在的 `tag_id`，给订阅源分配文件夹只接受已存在的 `folder_id` 或置空；标签名和文件夹名必须各自唯一，重名创建返回冲突错误；MVP 不得在分配流程中即时新建标签 / 文件夹。
- MVP 首版导航必须覆盖内容收件箱、稍后处理、收藏、已完成、订阅源、标签、搜索和设置；设置中可包含单人用户账户和鉴权配置，但不得引入社区、推荐或知识库等后置概念。
- 内容收件箱、稍后处理、收藏和已完成必须实现为同一内容条目列表的预设过滤视图，复用同一套查询、分页、排序、单值 AND 过滤和条目操作；MVP 不得为这些视图引入独立列表模型、专属工作流或仪表盘。
- 订阅源详情必须实现为同一内容条目列表预设 `feed_id` 过滤的视图，复用同一套查询、分页、排序和条目操作；`feed_id` 是单值过滤维度，可与其他单值过滤条件 AND 组合；MVP 不得为订阅源详情引入独立列表模型或订阅源级专属统计面板。
- 内容收件箱必须优先按内容条目发布时间排序，避免因抓取或导入时间污染信息流顺序。
- 首次抓取和 OPML 导入必须限制每个订阅源的历史内容数量，避免冷启动过慢、收件箱被历史内容淹没或自动摘要成本失控。
- 增量抓取必须在同一订阅源内避免重复内容条目；同源内必须按 feed item `guid`、`link`、`title + published_at` 稳定哈希的优先级生成 `dedupe_key`，并使用 `(feed_id, dedupe_key)` 唯一约束保证幂等；MVP 不要求跨订阅源识别或合并重复 / 相似内容。
- 增量抓取遇到已存在内容条目时必须跳过该条目，不更新其 feed 文本字段或 `available_text`；MVP 不做已有内容条目修订同步、摘要过期标记或重新导出提示。
- 产品规划必须先明确功能全集和 MVP / 后续迭代分层，再进入技术设计或实现。
- MVP 功能优先级必须先形成“获取内容 → 消费内容 → 整理内容 → 找回内容 → 摘要理解 → 外部沉淀”的闭环，再扩展自动摘要和批量导入。
- MVP 功能必须形成可验证的用户闭环，而不是只堆功能点。
- MVP 只需要接入结构化订阅来源：RSS feed URL、Podcast RSS feed URL 和 OPML 一次性批量导入。
- OPML 导入不得扩大内容模型边界；它只用于批量创建订阅源，不表示后续同步或复杂文件夹管理；MVP 导入时必须忽略 OPML 文件夹 / 分组层级。

## MVP Functional Scope

- 订阅源管理：添加、删除、查看 RSS 源和播客源；订阅源类型只支持 `rss` / `podcast`；添加订阅源时按规范化 feed URL 去重，并且只有首次抓取解析成功才创建订阅源；删除订阅源时同时删除该订阅源产生的历史内容条目及其摘要、播放进度、内容标记和标签关联等派生数据。
- 内容收件箱：聚合展示来自订阅源的新内容条目；作为预设过滤视图默认 `processing_status=unprocessed`；不接受关键词，默认按发布时间倒序，缺失或异常发布时间时回退到抓取时间；与搜索结果共享内容类型、处理状态、内容标记、标签和文件夹的单值 AND 过滤语义。
- 订阅源抓取：添加订阅源后立即抓取一次；首次抓取每个订阅源最多保存最近 50 条历史内容；支持手动刷新全部订阅源和手动刷新单个订阅源；支持一个全局后台定时抓取间隔，默认 60 分钟，可选关闭、30 分钟、60 分钟、6 小时或 24 小时；同一订阅源抓取互斥，若已有抓取进行中则新的同源抓取跳过；后续抓取只新增同一订阅源内未见过的新内容条目。
- 抓取状态：每个订阅源只保留当前 / 最后状态字段：`fetch_status`（`idle` / `fetching` / `success` / `failed`）、`fetch_started_at`、`last_fetched_at`、`last_fetch_error`；不提供抓取历史列表。
- 基础阅读：打开文本型内容条目，基础渲染 feed 提供的标题、正文、摘要、来源、发布时间；后端保留原始 feed 文本字段并生成 `available_text`；如果只有摘要，则提供打开原文链接；不持久化文本阅读位置。
- 基础收听：打开音频型内容条目，支持播放 / 暂停、进度条拖动、持久化播放进度、继续播放、15 秒快进 / 后退和倍速播放。
- 处理状态：内容条目默认未处理；使用者可手动标记为已完成，也可把已完成改回未处理；阅读 / 收听进度不自动改变处理状态。
- 稍后处理：把内容条目标记为稍后处理，可与主状态叠加；稍后处理视图是预设 `mark=later` 的过滤视图。
- 收藏：保存高价值内容条目，可与主状态叠加；收藏视图是预设 `mark=favorite` 的过滤视图。
- 标签：支持创建、重命名、删除标签；支持给单个内容条目添加 / 移除多个标签，分配只接受已存在的 `tag_id`；标签名唯一，重名创建返回冲突；支持按标签过滤内容条目；删除标签只移除标签及其关联，内容条目保留。
- 文件夹：支持创建、重命名、删除文件夹；支持给单个订阅源选择零个或一个文件夹，分配只接受已存在的 `folder_id` 或置空；文件夹名唯一，重名创建返回冲突；支持按文件夹过滤内容收件箱和搜索结果；删除文件夹只移除文件夹本身，原属于它的订阅源置为无文件夹并保留。
- 基础搜索：使用 SQLite FTS5 按关键词检索内容条目的标题、订阅源名称、`available_text` 和已生成 AI 摘要；有关键词时默认按相关性排序，相关性相同时按内容条目发布时间倒序；无关键词、只有过滤条件时按内容条目发布时间倒序；与内容收件箱共享按内容类型、处理状态、内容标记、标签和文件夹做单值过滤的 AND 语义；MVP 内容类型过滤只提供 `text` / `audio`；不做发布时间范围过滤、多选过滤、OR 条件组、保存搜索条件、中文分词插件、拼音搜索、模糊纠错或语义搜索。
- 单条手动 AI 摘要：使用者可对一个内容条目基于 `available_text` 手动生成固定结构化 Markdown 摘要；每个内容条目只保留一个当前摘要及 `none` / `pending` / `success` / `failed` / `insufficient_text` 状态；允许在 `none` / `failed` / `success` 状态触发，成功后覆盖当前摘要或错误；`pending` 时返回正在生成；`insufficient_text` 不可重试；文本不足时提示无法生成可靠摘要；失败后记录原因并允许使用者手动重试，但不自动重试。
- 自动 AI 摘要配置：使用者可打开或关闭自动摘要，并选择自动摘要范围为文本型内容条目、音频型内容条目或两者；自动摘要只对配置开启后新抓取入库且摘要状态为 `none` 的内容条目生效；不会自动覆盖 `success`，也不会自动重试 `failed` 或 `insufficient_text`；文本不足或生成失败时跳过并记录原因，不自动重试。
- AI 服务配置：设置页允许使用者配置 API Base URL、API Key 和模型名，供手动摘要和自动摘要共用；AI 服务协议只支持 OpenAI-compatible、非流式 Chat Completions；保存配置只做格式校验，不强制发起模型测试请求；提供可选测试配置动作；自动摘要开启前只要求配置字段存在，不要求最近测试成功；API Key 保存于单实例数据库并只在服务端使用；前端保存后只显示已配置状态，不回显完整密钥；拾流不提供内置官方服务端密钥。
- 单条 Obsidian 导出：把一个内容条目复制或下载为 Markdown，并以普通 Markdown 元信息包含标签和订阅源文件夹；导出不强制生成 AI 摘要，但始终包含 `AI 摘要` 区块，按当前摘要状态写入摘要内容或状态说明；只包含 `available_text` 前 2000 个字符作为可用文本摘录，截断时提示打开原文链接查看完整内容。
- 首版导航：提供内容收件箱、稍后处理、收藏、已完成、订阅源、标签、搜索、设置视图；内容收件箱、稍后处理、收藏和已完成均为同一内容列表的预设过滤视图（分别预设 `processing_status=unprocessed`、`mark=later`、`mark=favorite`、`processing_status=completed`）。
- 单人用户账户：首次访问通过一次性初始化流程创建唯一用户账户；之后只能登录已有账户；使用 `username + password` 登录，不要求邮箱、昵称或 profile；用户账户模型只保留鉴权必需字段：`id`、`username`、`password_hash`、登录失败计数、锁定到期时间、创建时间和更新时间；要求强密码输入（最少 12 个字符，不强制大小写 / 数字 / 符号组合）并使用 bcrypt（cost 12）保存密码哈希；登录会话采用 30 天有效的 JWT Bearer access token，API 请求使用 `Authorization: Bearer <token>`，JWT 签名密钥由使用者自行修改 yml 配置，手动退出由前端删除 token；支持已登录状态下修改密码；登录保护只按唯一用户账户记录失败次数和锁定到期时间，连续 5 次失败锁定 15 分钟，成功登录后清空失败计数；登录页不提供 Web 端忘记密码流程；首版不规划 CLI 重置密码能力，也不承诺产品内密码恢复；忘记密码只能通过备份恢复或重置实例数据后重新首次初始化；该账户用于保护 VPS / 自托管部署访问；不支持邮箱字段、昵称、profile、个人资料编辑、邮箱验证、邮件通知、邮箱找回密码、公开注册、多用户邀请、角色权限、多租户数据隔离、refresh token、服务端 session 表、JWT 密钥轮换、生产环境密钥强制校验、可配置 hash cost、Argon2、scrypt、2FA、TOTP、Passkey、硬件密钥、登录历史、安全审计页面、安全通知或设备列表。
- 后端 API：采用 REST JSON API，统一前缀 `/api/v1`；沿用 Nunu 模板统一响应结构 `{ "code": <int>, "message": "...", "data": ... }`，成功响应 HTTP 200 且 `code` 为 0，失败响应由业务 `code` / `message` 表达并用 HTTP status 表达错误类别；分页列表在 `data` 内包含 `items` 和 `page`；列表 API 使用 `page + pageSize` 页码分页（`page` 从 1 开始），默认 `pageSize=20`，最大 `pageSize=100`，返回 `page:{page,pageSize,total}`；OpenAPI / Swagger 只作为开发辅助，不作为产品功能。
- 本地 / VPS / 自托管单实例运行：主要通过 Docker Compose 部署同一 Nunu Go 后端镜像，并启动 `server` 与 `task` 两个服务共享 SQLite 数据 volume；`server` 运行 `cmd/server` 提供 HTTP API，`task` 运行 `cmd/task` 执行 Nunu `TaskServer` / `gocron` 定时任务；数据库结构变化通过 `cmd/migration` 调用 `golang-migrate/migrate` 执行成对 SQL migration 文件，并先于 `server` / `task` 正常运行；使用一个 SQLite 文件数据库保存订阅源、内容条目、处理状态、内容标记、摘要、用户账户与导出所需数据；保留 Nunu 的 `cmd/server`、`cmd/migration`、`cmd/task` 入口形态，但不引入 Redis、MySQL 或额外数据库服务；React 前端作为产品前端方向，但本次只讨论后端边界；源码本地运行只作为开发方式；应用不内置 HTTPS / TLS 或证书管理，公网部署必须由自管反向代理或平台提供 TLS；产品内不提供整库备份 / 恢复，部署文档说明备份数据库 / volume。
- OPML 导入：支持上传或粘贴 OPML，一次性批量导入 RSS / Podcast RSS 订阅源；只读取 feed URL，忽略原 OPML 文件夹 / 分组层级，不自动创建拾流文件夹或分配订阅源文件夹；导入时按规范化 feed URL 统计重复数量；每个新增订阅源必须成功抓取并解析 feed 后才创建；导入后显示成功、重复、失败数量，并对新增订阅源保存最近 50 条历史内容。

## MVP Functional Priority

1. 订阅源管理、订阅源抓取和内容收件箱：先形成稳定内容流。
2. 文本阅读、音频收听、处理状态、稍后处理和收藏：让使用者能真实消费并整理内容条目。
3. 基础搜索、标签过滤和文件夹过滤：让内容条目可重新找到。
4. 手动 AI 摘要和 AI 服务配置：建立拾流相对普通订阅阅读器的差异化理解能力。
5. Obsidian 单条 Markdown 导出：完成外部知识工具沉淀闭环。
6. 自动摘要：在抓取、AI 服务配置、摘要状态和失败处理稳定后实现。
7. OPML 导入：在单个订阅源添加与抓取稳定后实现批量导入。

## AI Summary Markdown Structure

- `TL;DR`：一句话总结。
- `要点`：3 到 5 条关键点。
- `对开发者 / 信息重度用户的价值`：说明为什么值得读 / 听，以及可能的行动或延伸思考。
- `原文信息`：来源和链接。

## Obsidian Export Markdown Fields

- 标题。
- 原文或原节目链接。
- 订阅源名称。
- 发布时间。
- 内容类型。
- 标签。
- 订阅源文件夹。
- AI 摘要区块：`success` 写当前摘要 Markdown；`none` / `pending` / `failed` / `insufficient_text` 写状态说明。
- 可用文本摘录：`available_text` 前 2000 个字符；截断时提示打开原文链接查看完整内容。
