# 拾流 Context

拾流是一个面向开发者与信息重度用户的订阅中心语境。本文只记录领域语言，不记录实现细节或产品规格。

## Language

**拾流 / Shiliu**:
面向开发者与信息重度用户的订阅中心产品。
_Avoid_: RSS 阅读器、播客播放器（这些只描述局部形态，不等同于完整产品）

**使用者**:
实际操作拾流来管理、消费和沉淀订阅内容的人。
_Avoid_: 用户账户、账号（这些表示身份系统对象，不等同于产品操作者）

**用户账户**:
用于拾流单人部署中的登录、会话和访问保护的身份对象；MVP 只支持一个用户账户，不表示多用户、团队成员或多租户数据归属体系。
_Avoid_: 用户（该词容易混淆产品受众和身份系统对象）、多用户账户、团队账号

**首次初始化**:
拾流单实例第一次被访问时创建唯一用户账户的设置流程；完成后不再提供公开创建用户账户的入口。
_Avoid_: 公开注册、邀请注册、多账号开通（这些暗示持续开放的账户创建能力）

**订阅中心**:
统一管理文本与音频内容来源，并让使用者在一个地方消费、整理和再利用内容的产品空间。
_Avoid_: 新闻客户端、播客客户端、知识库（这些词容易把范围收窄或偏离订阅管理）

**单实例**:
由一个使用者或一个受控环境使用的一份拾流运行实例，拥有一套订阅源、内容条目、本地数据边界和单人用户账户。
_Avoid_: 多租户、共享实例（这些暗示多个账户或多个互相隔离的数据空间）

**订阅源**:
使用者主动添加的结构化内容来源；一个订阅源会持续产生多个内容条目。
_Avoid_: 来源、频道、账号（这些词边界更宽，容易包含非订阅或非结构化输入）

**订阅源抓取**:
拾流从订阅源读取最新 feed 数据并生成或更新内容条目的动作。
_Avoid_: 同步（同步容易暗示多设备或双向数据一致性，不适合描述单向读取 feed）

**OPML 导入**:
使用 OPML 文档一次性批量创建订阅源的导入动作。
_Avoid_: OPML 同步、文件夹管理（这些暗示持续同步或复杂组织结构）

**内容条目**:
订阅源产生的一条可消费内容；文章、博客、新闻、播客单集和音频节目单集都是内容条目的不同类型。
_Avoid_: 文章、节目、Feed item（这些词只能覆盖部分内容类型或过于技术化）

**可用文本**:
订阅源数据中已经提供、可被拾流直接展示、搜索、摘要或导出的文本内容，包括标题、描述、摘要、正文片段、全文或 show notes。
_Avoid_: 原网页全文、阅读模式正文（它们需要额外抓取和抽取，不属于 feed 已提供文本）

**文本阅读**:
使用者消费文本型内容条目的基础阅读能力；MVP 只基础渲染可用文本并提供打开原文链接。
_Avoid_: 高亮、批注、阅读主题、字体设置（这些是深度阅读器能力，不是 MVP 文本阅读边界）

**内容收件箱**:
来自多个订阅源的新内容条目聚合入口，承载使用者的待消费信息流。
_Avoid_: 首页、Feed、时间线（这些词不能表达“待处理入口”的含义）

**处理状态**:
内容条目的主流程位置，表示使用者是否仍需要处理该内容条目；拾流使用未处理和已完成作为处理状态，且只能由使用者手动改变。
_Avoid_: 已读状态、播放完成状态（这些把消费进度误当成处理状态）

**消费进度**:
使用者阅读或收听内容条目时留下的位置记录；MVP 只持久化音频播放进度，不持久化文本阅读位置。
_Avoid_: 处理状态（消费进度不决定内容条目是否已完成）

**音频播放器**:
用于消费音频型内容条目的播放界面；MVP 支持播放 / 暂停、进度条拖动、继续播放、15 秒快进 / 后退和倍速播放。
_Avoid_: 播放队列、章节、离线下载（这些是后续播客体验能力，不是 MVP 播放器边界）

**内容标记**:
使用者叠加在内容条目上的意图或价值判断；稍后处理和收藏都是内容标记。
_Avoid_: 状态（状态应表示主流程位置，标记应允许叠加）

**标签**:
贴在内容条目上的自定义组织词，用来表达主题、用途或沉淀意图；一个内容条目可以有多个标签。MVP 中标签只支持创建、重命名、删除、单条分配和过滤。
_Avoid_: 文件夹、自动打标、批量打标（文件夹用于组织订阅源；自动化和批量能力不是 MVP 标签能力）

**文件夹**:
用于组织订阅源的自定义分组；一个订阅源最多属于一个文件夹。MVP 中文件夹只支持创建、重命名、删除、单条分配和过滤。
_Avoid_: 标签、嵌套文件夹、多文件夹归属（标签用于组织内容条目；嵌套和多归属不是 MVP 文件夹能力）

**AI 摘要**:
基于内容条目可用文本生成的固定结构化 Markdown 理解结果；当可用文本不足时，不应假装理解缺失内容。手动 AI 摘要是 MVP 必备动作。
_Avoid_: 播客转写、全文理解、纯段落摘要（这些暗示系统读取了音频或不可用正文，或削弱 Obsidian 沉淀价值）

**自动摘要**:
由使用者配置开启后，拾流在后台为符合内容类型范围的新内容条目生成 AI 摘要的能力；MVP 中只支持全局开关和内容类型范围。
_Avoid_: 强制摘要、批量摘要、规则引擎（自动摘要必须可配置；批量摘要和复杂规则是高级能力）

**AI 服务配置**:
使用者为 AI 摘要提供的外部模型服务连接信息；MVP 中拾流不提供官方托管 AI 配额。
_Avoid_: 官方服务端密钥、平台内置 AI 服务（这些暗示拾流承担公共成本、配额和滥用防护）

**基础搜索**:
通过关键词和少量结构化过滤条件重新找到内容条目的能力；它不表示向量语义搜索或问答式检索。
_Avoid_: 语义搜索、智能搜索（这些暗示更重的 AI 检索能力）

**Obsidian 导出**:
把单个内容条目及其摘要整理成可进入 Obsidian 的 Markdown 内容；MVP 导出包含标签和订阅源文件夹，但只作为普通 Markdown 元信息。
_Avoid_: Obsidian 同步、Obsidian 插件、frontmatter（这些表示更深的平台集成或模板约定，不等同于导出）

## Example Dialogue

产品：拾流不是单纯 RSS 阅读器，也不是单纯播客播放器；它是订阅中心。
开发：这里说的用户，是不是数据库里的用户账户？
产品：不是。我们用“使用者”指产品操作者；用户账户是单人部署中的登录和访问保护对象。
开发：那拾流 MVP 是多个人一起用一个公开站点吗？
产品：不是。MVP 是单实例产品，一套运行实例对应一套受控数据边界和一个用户账户。
开发：所以使用者可以在拾流里追踪博客文章、新闻和播客节目？
产品：对，使用者先添加订阅源；也可以通过 OPML 导入一次性批量创建订阅源。拾流通过订阅源抓取把新内容条目放进内容收件箱。
开发：文本内容会抓原网页全文吗？
产品：MVP 不会。拾流只使用订阅源提供的可用文本；原网页全文抓取后置。
开发：文本阅读会有高亮、批注或主题设置吗？
产品：MVP 不会。文本阅读只基础渲染可用文本并提供打开原文链接。
开发：文章和播客单集都是内容条目吗？
产品：对。它们的消费方式不同，但都可以被收藏、稍后处理、基础搜索、摘要或导出。
开发：标签和文件夹都能组织内容吗？
产品：不是。标签用于内容条目，文件夹用于订阅源；MVP 只做创建、重命名、删除、单条分配和过滤。内容收件箱可以按文件夹过滤来源。
开发：基础搜索是语义搜索吗？
产品：不是。MVP 的基础搜索指关键词搜索和基础过滤，语义搜索后置。
开发：收藏会不会和已完成冲突？
产品：不会。已完成是处理状态，收藏是内容标记，它们可以叠加。
开发：听完播客会自动变成已完成吗？
产品：不会。音频播放进度属于消费进度，只用于继续收听；处理状态必须由使用者手动改变。
开发：MVP 播放器会有队列、章节和下载吗？
产品：不会。这些是迭代功能；MVP 音频播放器只覆盖基础播放、进度恢复、15 秒快进 / 后退和倍速。
开发：音频型内容条目的 AI 摘要会理解音频内容吗？
产品：MVP 不会。AI 摘要只基于已有文本；如果节目描述或 show notes 不足，就不能生成可靠摘要。
开发：AI 摘要是随便一段话吗？
产品：不是。MVP 的 AI 摘要是固定结构化 Markdown，包含 TL;DR、要点、对开发者 / 信息重度用户的价值和原文信息。
开发：抓取到新内容后会自动摘要吗？
产品：可以，但必须由使用者配置开启；MVP 只支持全局开关和内容类型范围，手动 AI 摘要始终可用。
开发：拾流会内置官方 AI 服务端密钥吗？
产品：不会。AI 服务配置由使用者提供，拾流 MVP 不承担官方托管 AI 配额或计费。
开发：Obsidian 导出会直接同步到使用者 vault 吗？
产品：MVP 不会。Obsidian 导出指单条 Markdown 复制或下载，会包含标签和订阅源文件夹作为普通元信息，但不做 frontmatter 或深度同步。
